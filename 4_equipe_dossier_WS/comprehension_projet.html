01 — Point de départ

C'est quoi le problème qu'on résout ?

Avant de parler de code, il faut comprendre pourquoi ce projet existe et ce qu'il apporte concrètement.

Aujourd'hui, les restaurants de Dakar, les boutiques Jumia et les vendeurs Amazon reçoivent des centaines d'avis clients sur plusieurs plateformes différentes. Le problème : personne ne peut lire 2 000 avis manuellement pour comprendre ce que pensent vraiment les clients.

Review Analyzer résout ce problème en automatisant toute la chaîne : collecter ces avis → les analyser → les présenter de façon lisible. Un restaurateur peut ainsi savoir en quelques secondes que 82% de ses clients sont satisfaits, que le mot-clé "service lent" revient souvent dans les avis négatifs, et que sa note a baissé ces 3 derniers mois — sans lire un seul avis.

🎯

Pour les restaurateurs

Comprendre la réputation de leur établissement en un coup d'œil, sans passer des heures sur TripAdvisor.

🛍️

Pour les e-commerçants

Identifier les points faibles de leurs produits sur Amazon et Jumia grâce aux mots-clés négatifs récurrents.

📊

Pour l'équipe

Démontrer la maîtrise d'un pipeline complet : scraping → NLP → API → mobile. Un projet de bout en bout.

↓

Principe clé : chaque couche est indépendante. Le scraping ne sait pas que le NLP existe. Le NLP ne sait pas que l'API existe. Ils communiquent uniquement via la base de données et les fichiers JSON. Ça permet de modifier une couche sans casser les autres.

03 — Collecte des données

Comment on a scrappé chaque plateforme

Chaque plateforme a sa propre structure HTML, ses propres protections anti-bot, et nécessite une stratégie d'extraction différente.

Pourquoi Selenium et pas Scrapy ?

La première intuition était d'utiliser Scrapy, un framework Python rapide et populaire pour le scraping. Mais TripAdvisor, Amazon et Google Maps chargent leurs contenus dynamiquement via JavaScript — le HTML initial ne contient pas les avis. Scrapy lit uniquement le HTML statique, donc il ne voit rien.

Selenium résout ce problème en contrôlant un vrai navigateur Chrome qui exécute le JavaScript exactement comme un humain. On voit ce que l'utilisateur verrait.

⚙️

Mode furtif Chrome : Chrome est lancé avec des options spéciales pour ressembler à un vrai utilisateur — suppression du flag navigator.webdriver, user-agent Chrome 144 réaliste, profil temporaire dans C:\Temp\chrome_selenium. Sans ça, TripAdvisor détecte le bot en quelques secondes.

TripAdvisor — 1 917 avis

1
Collecte des URLs des restaurants

Le spider ouvre la page liste "restaurants Dakar" sur TripAdvisor, accepte les cookies automatiquement, puis extrait tous les liens contenant Restaurant_Review. Chaque lien est normalisé (suppression du #REVIEWS) et dédupliqué dans un set Python.

2
Extraction des avis par restaurant

Pour chaque restaurant, le spider récupère : le nom via le premier <h1> non vide (sélecteur agnostique aux classes CSS changeantes), les cartes d'avis via div[data-automation='reviewCard'], le texte complet (clic sur "Plus" si tronqué), la date et l'auteur.

3
Extraction de la note — 4 stratégies en cascade

Les notes TripAdvisor sont dans des SVG avec des classes CSS dynamiques qui changent à chaque mise à jour du site. On utilise 4 méthodes dans l'ordre : aria-label sur le SVG → balise <title> → attribut data-rating → requête JavaScript directe sur le DOM.

4
Pagination automatique — 3 conditions d'arrêt

L'URL de la page suivante est construite par injection d'un offset (-or10-, -or20-...). On s'arrête si : la page est vide (0 avis), le total déclaré par TripAdvisor est atteint, ou la page contient moins de 10 avis (dernière page). Limite absolue : 50 pages par restaurant.

Amazon — 148 avis

Le spider Amazon construit l'URL des avis à partir de l'ASIN du produit : amazon.fr/product-reviews/ASIN/?sortBy=recent. Il attend 12 secondes le chargement complet, détecte automatiquement les CAPTCHA, extrait les cartes via [data-hook='review'] et pagine jusqu'à 5 pages via le bouton li.a-last a.

⚠️

Problème rencontré : Le sélecteur de nom de produit récupérait le texte des avis au lieu du vrai nom — résultat : les produits s'appelaient "socle", "Bonne bouilloire", "C'est un téléphone super...". Corrigé en post-traitement via fix_amazon_product_names.py qui mappe les faux noms vers les vrais.

Google Maps & Jumia — 17 avis

Ces plateformes ont été scrapées par d'autres membres de l'équipe qui ont fourni leurs données en format JSON. Ces fichiers contenaient des caractères corrompus (Ã© au lieu de é) — une confusion encodage latin-1/UTF-8 — corrigée automatiquement à l'import.

Le problème du blocage anti-bot

Après avoir scrapé 6 restaurants intensivement (des centaines de pages), TripAdvisor a bloqué les requêtes par détection IP. La solution : supprimer le profil Chrome entre les sessions (Remove-Item -Recurse -Force "C:\Temp\chrome_selenium") et espacer les sessions de 4 à 6 heures. Résultat : seuls 6 restaurants sur 35 ont été scrapés.

04 — Base de données

Comment les données sont stockées

Toutes les données, quelle que soit leur plateforme d'origine, sont stockées dans une base SQLite unique avec un schéma unifié.

Pourquoi un schéma unifié ?

TripAdvisor, Amazon, Google Maps et Jumia ont des formats d'avis très différents. Plutôt que de créer une table par plateforme, on a conçu une seule table reviews avec un champ platform qui identifie l'origine. Ça simplifie drastiquement les requêtes : pour analyser tous les avis, on fait une seule requête sans jointures complexes.

database/schema.pyPython
class Review(Base):
    __tablename__ = "reviews"

    # ── Identification ────────────────────────────────────────
    id           = Column(String(36), primary_key=True, default=lambda: str(uuid4()))
    product_name = Column(String(300))    # "Restaurant Lagon 1", "iPhone 17"...
    platform     = Column(Enum("amazon", "jumia_sn", "googlemaps", "tripadvisor"))

    # ── Données brutes (collectées par le spider) ─────────────
    rating       = Column(Float)            # 1.0 à 5.0
    comment_text = Column(Text)             # texte complet de l'avis
    comment_date = Column(String(50))      # date brute telle qu'affichée
    author       = Column(String(150))    # nom de l'auteur
    url_source   = Column(String(500))    # URL d'origine
    scraped_at   = Column(DateTime, default=datetime.utcnow)

    # ── Résultats NLP (remplis par nlp_pipeline.py) ───────────
    language        = Column(String(10))   # 'fr', 'en', 'wo'...
    sentiment       = Column(Enum("positive", "negative", "neutral"))
    sentiment_score = Column(Float)         # score de confiance : 0.0 → 1.0
    keywords        = Column(JSON)          # ["service lent", "cuisine excellente"]

Comment SQLAlchemy simplifie le travail

SQLAlchemy est un ORM (Object-Relational Mapper). Au lieu d'écrire du SQL brut, on manipule des objets Python. Pour insérer un avis, on crée un objet Review et on fait db.add(review); db.commit() — SQLAlchemy génère le SQL automatiquement. Et si on change de base (SQLite → PostgreSQL), on change juste une ligne dans .env.

Le workflow import/export JSON

Comme l'équipe travaille sur des machines différentes, les données circulent en JSON — un format universel lisible par tous. Voici le flux :

1
Scraping → SQLite directement

Les spiders TripAdvisor et Amazon insèrent les avis directement dans la base SQLite locale via SQLAlchemy, en temps réel pendant le scraping.

2
SQLite → JSON (export)

export_tripadvisor.py et export_amazon.py lisent la base et génèrent des fichiers JSON uniformisés dans data/raw/. Ces fichiers peuvent être partagés avec l'équipe.

3
JSON → SQLite (import unifié)

import_reviews.py accepte plusieurs fichiers JSON en argument (Amazon, Google Maps, Jumia...), corrige l'encodage, vérifie les doublons via 4 critères (auteur + produit + date + plateforme), et insère les nouveaux avis.

4
Nettoyage des doublons

clean_amazon_duplicates.py supprime les enregistrements avec les anciens noms de produits incorrects. A supprimé 149 doublons pour passer de 297 à 148 avis Amazon propres.

05 — Intelligence artificielle

Comment fonctionne la pipeline NLP

Le NLP (Natural Language Processing) est le cœur analytique du projet. C'est ce qui transforme un texte brut en information structurée et exploitable.

Les 3 analyses effectuées sur chaque avis

🌍

1. Détection de langue

Identifie si l'avis est en français, anglais, ou autre. Important car les modèles de sentiment ne sont pas les mêmes selon la langue.

❤️

2. Analyse de sentiment

Classifie l'avis en positive, negative ou neutral. C'est l'analyse principale — elle répond à "le client est-il satisfait ?".

🏷️

3. Extraction de mots-clés

Identifie les thèmes récurrents : "service lent", "cuisine excellente", "rapport qualité/prix", etc. Répond à "pourquoi est-il satisfait ou pas ?".

Les modèles utilisés

distilcamembert-base-sentiment
Pour les avis en FRANÇAIS

CamemBERT est l'équivalent français de BERT — un modèle Transformer pré-entraîné sur des milliards de mots français. La variante "distil" est plus légère (~250MB) et aussi précise.

Français Avis consommateurs cmarkea

Retourne des labels "1 star" à "5 stars" — convertis ensuite en positive/negative/neutral via un mapping.

twitter-roberta-base-sentiment
Pour les avis en ANGLAIS et autres

RoBERTa est une version optimisée de BERT, fine-tunée sur des données Twitter multilingues (~500MB). Gère directement les labels positive/negative/neutral.

Anglais Multilingue cardiffnlp

Retourne directement negative/neutral/positive avec un score de confiance entre 0 et 1.

paraphrase-multilingual-MiniLM-L12-v2
Pour les MOTS-CLÉS — via KeyBERT

KeyBERT utilise un modèle de sentence embeddings pour représenter les phrases et les mots-clés sous forme de vecteurs numériques, puis trouve les mots dont le vecteur est le plus proche du vecteur de l'avis entier. Ce modèle MiniLM gère le français, l'anglais et 50+ langues.

FR + EN + 50 langues sentence-transformers unigrammes + bigrammes

Comment fonctionne la vectorisation

La vectorisation est la méthode qui permet à un modèle de "comprendre" du texte. En voici le principe simplifié :

1
Tokenisation

Le texte est découpé en tokens — petits morceaux de mots. "Restaurant" devient par exemple ["Rest", "##aurant"]. CamemBERT accepte jusqu'à 512 tokens (~800 mots). Les avis plus longs sont tronqués à 1 000 caractères avant tokenisation.

2
Embedding — conversion en vecteurs

Chaque token est converti en un vecteur de 768 nombres (pour BERT de base). Ces vecteurs encodent le sens du mot dans son contexte — "service" dans "service excellent" aura un vecteur différent de "service" dans "service catastrophique".

3
Classification par le modèle

Le modèle prend les vecteurs en entrée et produit des scores pour chaque classe (positive, negative, neutral). La classe avec le score le plus élevé est choisie. Ce score devient le sentiment_score — la confiance du modèle.

4
Extraction de mots-clés (KeyBERT)

KeyBERT encode l'avis entier en un vecteur global, puis encode chaque candidat mot-clé (unigramme ou bigramme) en vecteur. Les mots-clés retenus sont ceux dont le vecteur a la plus haute similarité cosinus avec le vecteur de l'avis. use_maxsum=True garantit la diversité des mots-clés retenus.

La correction du seuil "neutral"

Un problème courant en NLP sur avis : les modèles surclassifient en "neutral" les avis courts ou ambigus. On a mis en place une correction post-modèle :

nlp_pipeline.py — Correction du seuilPython
# Si le modèle dit "neutral" avec une confiance < 70%,
# on corrige en utilisant la note (rating) comme signal fort
if label == "neutral" and score < 0.70:
    if rating is not None:
        if rating >= 4:    label = "positive"   # 4-5★ → positif
        elif rating <= 2: label = "negative"   # 1-2★ → négatif
        else:            label = "positive"   # 3★ → positif par défaut
    else:
        label = "positive"  # sans note → positif par défaut

Ce seuil de 0.70 a été calibré empiriquement — validé à F1 Weighted = 87.4% sur les données d'évaluation.

Pourquoi F1 Weighted et pas la simple accuracy ?

Les données sont fortement déséquilibrées : environ 84% des avis sont positifs (les gens qui ont eu une mauvaise expérience écrivent rarement). Une simple accuracy serait trompeuse — un modèle qui dit "positive" tout le temps aurait 84% d'accuracy mais serait inutile. Le F1 Weighted pondère chaque classe par son nombre d'exemples réels, ce qui est la métrique standard en NLP sur données déséquilibrées.

06 — Exposition des données

Comment fonctionne l'API

L'API est le pont entre la base de données et l'application mobile. Elle transforme les données SQLite en réponses JSON accessibles via HTTP.

Pourquoi FastAPI et pas Flask ?

FastAPI présente trois avantages décisifs pour ce projet. Premièrement, il génère automatiquement une documentation Swagger interactive sur /docs — pratique pour tester les endpoints sans Postman. Deuxièmement, il utilise Pydantic pour valider automatiquement les données (types, valeurs obligatoires, plages autorisées). Troisièmement, il est nativement asynchrone et bien plus rapide que Flask pour des API à fort trafic.

Les schemas Pydantic — api/schemas.py

Les schemas définissent exactement ce que l'API retourne. Si la base contient un champ interne qu'on ne veut pas exposer, on ne le met pas dans le schema. Ils servent aussi de validation : si un champ est déclaré float et que la base retourne autre chose, FastAPI lève une erreur claire.

api/schemas.py — Exemple de schemaPython
class ReviewOut(BaseModel):
    id:              str
    product_name:    str
    platform:        str
    rating:          Optional[float]
    comment_text:    str
    comment_date:    Optional[str]
    author:          Optional[str]
    language:        Optional[str]
    sentiment:       Optional[str]   # "positive" | "negative" | "neutral"
    sentiment_score: Optional[float] # 0.0 → 1.0
    keywords:        Optional[list[str]]

    class Config:
        from_attributes = True  # lit directement depuis l'objet SQLAlchemy

Les endpoints disponibles

GET
/reviews
Liste paginée des avis. Filtres : platform, sentiment, language, search, min_rating, max_rating, limit, offset. Retourne le total et les résultats.
GET
/reviews/{id}
Détail complet d'un avis par son UUID. Retourne toutes les données incluant les résultats NLP.
GET
/stats
Statistiques globales : total des avis, note moyenne, répartition par sentiment, détail par plateforme avec positifs/négatifs.
GET
/score?product=Lagon
Score de réputation d'un restaurant ou produit spécifique. Retourne le % d'avis positifs, la note moyenne, et les top 10 mots-clés. C'est l'endpoint principal pour l'app mobile.
GET
/keywords
Top mots-clés les plus fréquents. Filtrable par plateforme et sentiment. Utile pour afficher un nuage de mots dans l'app.
GET
/suggestions?query=lag
Autocomplétion pour la barre de recherche. Retourne les noms de produits/lieux qui contiennent la chaîne recherchée.

07 — Application mobile

L'application Flutter — à quoi elle sert

Flutter est le framework de Google pour créer des applications mobiles en Dart qui tournent à la fois sur Android et iOS avec un seul code source.

Quel est son intérêt dans ce projet ?

La pipeline NLP produit des données structurées (sentiments, scores, mots-clés) stockées en base. L'API les expose via HTTP. Mais ces données restent inutiles pour un utilisateur non technique s'il n'y a pas d'interface pour les consulter.

L'application Flutter est la couche de présentation — ce que voit l'utilisateur final. Elle consomme l'API REST et transforme des données JSON en interface visuelle intuitive. Un restaurateur qui n'a jamais entendu parler de JSON peut ainsi consulter la réputation de son établissement depuis son téléphone.

📊

Dashboard de réputation

Affiche le score global d'un restaurant (% positifs, note moyenne) avec un graphique de distribution des sentiments.

🔍

Recherche avec autocomplétion

Barre de recherche connectée à l'endpoint /suggestions pour trouver rapidement un restaurant ou produit.

💬

Liste des avis filtrés

Affiche les avis avec leur sentiment coloré (vert/rouge/gris) et leurs mots-clés. Filtrable par sentiment.

☁️

Nuage de mots-clés

Visualise les thèmes récurrents grâce à l'endpoint /keywords — les mots les plus fréquents apparaissent plus grands.

Comment Flutter communique avec l'API

Flutter utilise le package http de Dart pour faire des requêtes HTTP vers l'API FastAPI. Le JSON retourné est désérialisé en objets Dart via des classes modèles. Voici le principe :

lib/main.dart — Appel APIDart
// Récupère le score de réputation d'un restaurant
Future<ScoreData> fetchScore(String productName) async {
  final response = await http.get(
    Uri.parse('http://localhost:8000/score?product=$productName'),
  );

  if (response.statusCode == 200) {
    return ScoreData.fromJson(jsonDecode(response.body));
  } else {
    throw Exception('Erreur API : ${response.statusCode}');
  }
}

// Affiche le résultat dans un widget
FutureBuilder<ScoreData>(
  future: fetchScore('Lagon'),
  builder: (context, snapshot) {
    if (snapshot.hasData) {
      return Text('Score : ${snapshot.data!.reputationScore}%');
    }
    return CircularProgressIndicator();
  },
)
📱

Un code, deux plateformes : le même code Dart tourne sur Android et iOS sans modification. Pour un prototype ou un projet académique, Flutter permet de livrer une vraie application mobile sans doubler le travail de développement.

08 — Vue d'ensemble

Le flux de données complet

Voici ce qui se passe exactement de bout en bout, de la page web TripAdvisor jusqu'à l'écran du téléphone de l'utilisateur.

1
Le spider ouvre Chrome et navigue sur TripAdvisor

Chrome est lancé en mode furtif. Le spider accepte les cookies, collecte les URLs des restaurants, puis pour chaque restaurant scrape toutes les pages d'avis. Chaque avis est inséré immédiatement en base SQLite.

2
Les avis des autres équipes sont importés

import_reviews.py fusionne les fichiers JSON Google Maps et Jumia dans la même base SQLite, en corrigeant l'encodage et en évitant les doublons.

3
La pipeline NLP analyse les 2 082 avis

nlp_pipeline.py lit chaque avis, détecte sa langue, choisit le bon modèle (CamemBERT pour le français, RoBERTa sinon), calcule le sentiment et le score de confiance, extrait les mots-clés, puis écrit les résultats dans les colonnes NLP de la base.

4
L'API FastAPI expose les données

Le serveur uvicorn démarre. L'API lit la base SQLite et répond aux requêtes HTTP. Elle sérialise les objets SQLAlchemy en JSON validé par Pydantic.

5
Flutter affiche les résultats

L'app mobile fait une requête GET /score?product=Lagon, reçoit le JSON, le désérialise en objet Dart, et affiche le score de réputation, les sentiments et les mots-clés à l'utilisateur.

✅

Ce que ce projet démontre : la maîtrise d'une stack technique complète — Selenium pour le scraping, SQLAlchemy pour la persistance, HuggingFace Transformers pour le NLP, FastAPI pour l'API REST, et Flutter pour le mobile. Chaque composant est professionnel et documenté.
