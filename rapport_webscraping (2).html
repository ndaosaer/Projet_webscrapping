01
Introduction

Objectif du projet

L'objectif principal de ce projet est de construire un pipeline complet de collecte d'avis clients en ligne, depuis le scraping automatisé jusqu'au stockage structuré en base de données, en vue d'une analyse NLP (traitement du langage naturel) et de sentiment analysis.

Le périmètre initial se concentre sur les restaurants de Dakar, Sénégal via TripAdvisor et Google Maps, avec une extension aux produits e-commerce via Jumia et Amazon. L'enjeu est de centraliser des données hétérogènes provenant de sources très différentes dans un schéma unifié exploitable.

🎯

Collecte automatisée

Scraper sans intervention manuelle les avis sur TripAdvisor, Google Maps, Jumia et Amazon.

🗄️

Base unifiée

Centraliser toutes les sources dans une seule base SQLite avec un schéma commun et extensible.

🧠

Prêt pour le NLP

Préparer les données pour l'analyse de sentiments, extraction de mots-clés et classification.

📊

Insights actionnables

Produire des visualisations sur la réputation des établissements et des produits.

03
Scraping

Spider TripAdvisor — Restaurants Dakar

Le spider principal utilise Selenium avec Chrome contrôlé en mode furtif pour collecter les avis des restaurants de Dakar, page par page, restaurant par restaurant.

Étape 1

Chrome en mode furtif

Chrome lancé avec suppression des flags d'automatisation, user-agent Chrome 144 réaliste, désactivation de navigator.webdriver via JavaScript, et profil temporaire dans C:\Temp\chrome_selenium.

Étape 2

Collecte des URLs (35 restaurants)

Ouverture de la page liste TripAdvisor Dakar, acceptation automatique des cookies RGPD, extraction de tous les liens Restaurant_Review. URLs normalisées (suppression du fragment #REVIEWS) pour éviter les doublons.

Étape 3

Extraction des données par restaurant

Pour chaque restaurant : nom (premier H1 non vide), cartes d'avis (div[data-automation='reviewCard']), note (système multi-fallback), texte complet (clic sur "Plus" si tronqué), date et auteur. Chaque avis inséré avec commit immédiat.

Étape 4

Pagination automatique

10 avis par page. L'URL suivante est construite par injection de l'offset (-or10-, -or20-...). 3 conditions d'arrêt indépendantes : page vide, total déclaré atteint, ou dernière page (<10 avis). Limite de sécurité : 50 pages max.

Étape 5

Export JSON (1 917 avis)

Script export_tripadvisor.py exporte tous les avis de SQLite vers tripadvisor_reviews.json au format unifié, identique aux autres plateformes.

Extraction de la note — Système multi-stratégie

TripAdvisor stocke les notes dans des SVG avec des classes CSS dynamiques changeant à chaque déploiement. Quatre stratégies de fallback sont appliquées en cascade :

tripadvisor_spider.py — _extract_rating()
# Stratégie 1 : aria-label sur SVG → "4 sur 5 bulles"  ← le plus fiable
svg = card.find_element(By.CSS_SELECTOR, "svg[aria-label]")
aria = svg.get_attribute("aria-label")
return float(aria.split()[0].replace(",", "."))

# Stratégie 2 : balise <title> dans n'importe quel SVG
if "sur" in title_text: return float(title_text.split()[0])

# Stratégie 3 : attribut data-rating ou data-value
val = elem.get_attribute("data-rating")

# Stratégie 4 : lecture du DOM via JavaScript
result = driver.execute_script(
    "return arguments[0].querySelector('svg title').textContent", card

04
Débogage

Problèmes rencontrés & solutions

Sept problèmes distincts ont été identifiés et résolus lors du développement et de l'exécution du spider TripAdvisor.

Problème #1

Spider non trouvé par Scrapy

KeyError: 'Spider not found: tripadvisor'. TripAdvisor charge ses avis en JavaScript dynamique — Scrapy seul ne peut pas les voir.

Solution

Migration vers Selenium pur

Abandon de Scrapy. Utilisation de Selenium + Chrome complet qui exécute le JavaScript comme un vrai navigateur.

Problème #2

Notes toutes à 0.0⭐

Sélecteur svg.evwcZ ne retournait rien — les classes CSS de TripAdvisor sont générées dynamiquement et changent à chaque déploiement.

Solution

Sélecteur d'attribut + 4 fallbacks

Remplacement par svg[aria-label], attribut sémantique stable. Ajout de 3 stratégies de fallback pour résister aux futurs changements.

Problème #3

URLs dupliquées dans la liste

La page liste renvoyait des liens avec et sans fragment #REVIEWS, causant le scraping en double des mêmes restaurants.

Solution

Normalisation avant déduplication

Méthode _normalize_url() : href.split("#")[0] avant ajout au set seen.

Problème #4

Erreur SQLite : unable to open database file

En lançant depuis scraping/spiders/, SQLite cherchait database/reviews.db dans ce sous-dossier plutôt qu'à la racine du projet.

Solution

Fix du répertoire de travail (CWD)

Ajout de os.chdir() en tête de chaque script pour forcer le CWD à la racine, quel que soit l'endroit de lancement.

Problème #5

Boucle de pagination infinie

Le bouton "Page suivante" était détecté comme toujours actif — les sélecteurs CSS ne matchaient pas. Le spider chargeait des pages indéfiniment.

Solution

3 conditions d'arrêt indépendantes

Arrêt si : page vide (0 avis), total déclaré atteint, ou <10 avis sur la page. Filet de sécurité absolu : 50 pages max par restaurant.

Problème #6

Blocage anti-bot TripAdvisor

Après scraping intensif (30+ restaurants, des centaines de pages), TripAdvisor a bloqué les requêtes. Pages vides, titre générique, aucun élément trouvé.

Solution

Reset du profil Chrome

Suppression de C:\Temp\chrome_selenium entre les sessions. Pauses aléatoires maintenues : 4–8s entre restaurants, 3–6s entre pages.

Problème #7

Classes CSS du H1 changées

Après reset du profil Chrome, les classes du titre H1 avaient changé. Le sélecteur codé en dur h1.biGQs._P.fiohW.eIegw ne matchait plus.

Solution

Sélecteur agnostique aux classes

Récupération de tous les <h1> de la page, sélection du premier avec texte non vide. Résistant à tous les changements CSS futurs.

05
Scraping

Spider Amazon.fr

Le spider Amazon scrape les pages d'avis de produits spécifiques à partir de leurs URLs. Trois produits ont été ciblés : un smartphone, une bouilloire et un support de casque.

1

Construction de l'URL d'avis

Extraction de l'ASIN depuis l'URL produit (/dp/ASIN/) et construction de l'URL dédiée aux avis : amazon.fr/product-reviews/ASIN/?sortBy=recent.

2

Chargement avec attente longue

Attente fixe de 12 secondes et scroll pour forcer le chargement complet. Détection automatique de CAPTCHA avec pause manuelle si nécessaire.

3

Extraction des avis

Sélecteur [data-hook='review'] pour les cartes. Note via i[data-hook='review-star-rating']. Texte = titre + " : " + corps de l'avis.

4

Pagination (max 5 pages)

Clic sur li.a-last a pour accéder à la page suivante. Arrêt automatique si le bouton est absent ou si la page est vide.

5

Résultat

148 avis collectés sur 3 produits : Apple iPhone 17 256 Go, Philips Bouilloire HD9350/90 et Alyvisun Support Casque.

Problème : noms de produits incorrects

Le sélecteur _get_product_name() ne trouvait pas le nom du produit et récupérait à la place le titre du premier avis ou un mot isolé du contenu : "socle", "Bonne bouilloire", "C'est un téléphone super je recommande vivement."

⚠️

Le spider amazon_spider.py n'ayant pas été modifié (contrainte du projet), la correction a été appliquée en post-traitement sur le fichier JSON exporté.

06
Intégration

Consolidation de toutes les sources

Une fois les spiders exécutés et les JSON des équipes partenaires reçus, quatre étapes ont permis de consolider l'ensemble des données dans une base SQLite unique et propre.

Étape 1 — Export des données scrapées

SQLite → JSON

Les avis collectés via Selenium (TripAdvisor et Amazon) étaient en base SQLite. Deux scripts les exportent en JSON pour uniformiser le format avec les fichiers reçus des autres équipes (Google Maps, Jumia).

Étape 2 — Correction de l'encodage

Réparation du latin-1 / UTF-8

Les fichiers Jumia et Google Maps contenaient des caractères corrompus : Ã© au lieu de é, â€™ au lieu de '. Le script d'import corrige automatiquement via text.encode('latin-1').decode('utf-8').

Étape 3 — Import unifié

JSON → SQLite (toutes plateformes)

Le script import_reviews.py accepte plusieurs fichiers JSON en argument. Il valide la plateforme, corrige l'encodage, vérifie les doublons (auteur + produit + date + plateforme) et insère les nouveaux avis.

Étape 4 — Nettoyage des doublons Amazon

Suppression des anciens enregistrements

Le spider Amazon avait inséré 149 avis avec les anciens noms incorrects. Après import du JSON corrigé, la base contenait 297 avis Amazon (doublons). Le script clean_amazon_duplicates.py a supprimé les 149 enregistrements avec anciens noms, ramenant à 148 avis propres.

✅

Résultat : 149 doublons supprimés (49 "socle" + 50 "Bonne bouilloire" + 50 "C'est un téléphone..."). Base finale : 2 082 avis propres sur 4 plateformes.

Scripts utilitaires & commandes

extract_card.py

Diagnostic : ouvre une page TripAdvisor, teste les sélecteurs CSS, sauvegarde le HTML complet. Utilisé pour identifier les bons sélecteurs après blocage.

Exporte les avis TripAdvisor de SQLite vers JSON. Résumé par restaurant. Fix CWD intégré.

Exporte les avis Amazon de SQLite vers JSON. Même format que TripAdvisor. Résumé par produit.

📥

Import universel : plusieurs fichiers JSON, toutes plateformes, correction encodage automatique, déduplication quadruple.

✏️

fix_amazon_product_names.py

Corrige les noms de produits incorrects dans amazon_reviews.json via un mapping ASIN → vrai nom. Exécuté deux fois (iPhone ajouté en second passage).

🧹

clean_amazon_duplicates.py

Supprime les enregistrements Amazon avec anciens noms incorrects. A supprimé 149 doublons pour revenir à 148 avis propres.

Prochaines étapes

Analyse de sentiments (NLP)

Appliquer un modèle sur comment_text. Remplir les colonnes sentiment, sentiment_score, language et keywords déjà prévues dans le schéma. Candidats : CamemBERT, TextBlob, ou API Anthropic.

Phase 3 — Moyen terme

Dashboard de visualisation

Interface web pour explorer les résultats : distribution des notes, évolution temporelle des sentiments, nuage de mots-clés, comparaison entre restaurants et plateformes.

Phase 4 — Long terme

Compléter le scraping TripAdvisor

Relancer le spider pour les restaurants 7 à 35. Sessions espacées de 4–6h, reset du profil Chrome entre chaque session, VPN si nécessaire.

Phase 5 — Long terme

Migration PostgreSQL

Pour un accès multi-utilisateurs ou un déploiement en production, changer DATABASE_URL dans .env. Aucune modification de code nécessaire grâce à SQLAlchemy.
